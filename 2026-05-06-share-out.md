# Social Scanner — what it is, what it found

## What it is

A local-first scanner that pulls posts from Reddit, Hacker News, and Bluesky every hour, classifies them via Claude Haiku for reactive WebCompat relevance, cross-references each finding against Bugzilla and webcompat/web-bugs to identify what's already on file, and writes dated markdown reports grouped by affected domain. Built in one day.

It augments BSR and webcompat.com with the informal channels users actually post to. Scope: any current Firefox problem, including site-specific breakage, platform regressions (crashes, freezes, slow), and broken extensions. Out of scope: feature requests, opinions, comparisons.

## What I tested

A backfill spanning 2026-02-09 to 2026-05-06 (about three months). Numbers below are from the current SQLite state, not in-flight estimates.

| | Count |
|---|---:|
| Posts pulled across the three sources | 2,350 |
| Posts classified relevant | **155** |
| ↳ webcompat with a domain | 65 |
| ↳ webcompat without a clear domain | 13 |
| ↳ Firefox platform issues (crashes, freezes, perf, UI) | 74 |
| ↳ misc / uncategorized | 3 |

Cost: ~$0.70 in Anthropic API tokens for the entire backfill (classification, bug matching, prompt caching).

## Headline finding

After cross-referencing every webcompat-with-domain item against existing bugs and using an LLM to compare each user post to each candidate bug summary:

| | Items |
|---|---:|
| Webcompat items with a domain | 65 |
| ↳ posts with at least one likely-match bug on file | **8** |
| ↳ posts with no likely match found | **57** |

The 8 "matches" are concentrated: 5 of them are different Bluesky posts all mapping to the same umbrella bug (wc#206744 "bsky.app - site is not usable"), 2 are claude.ai posts mapping to similar umbrella bugs, and 1 is the drive.google.com folder-upload crash matching BMO#2036974 verbatim.

So in practice: **57 user-reported web-compat issues with a clear domain, no clear matching bug filed.** Plus 13 more without a clear domain. That's the actionable bucket.

## Concrete examples worth a look

The strongest demonstration that the cross-reference is working:

- **drive.google.com folder-upload tab crash.** User on r/firefox describes uploading a folder with subfolders crashing the tab. The matcher correctly identifies BMO#2036974 ("drive.google.com - Uploading a folder with subfolders crashes the tab") as a verbatim match. Confirms impact on a tracked bug.
- **YouTube RAM/CPU consumption.** Two user reports describe YouTube using 1.5–7.5 GB RAM and freezing. The cross-reference returned 3 same-domain bugs (Download missing, Screenshot button, Drag-drop wording) and the matcher correctly flagged all 3 as `unrelated` ("different feature/symptom"). Strong signal that this RAM/freeze issue does not have a matching bug filed.
- **Facebook Marketplace 20+ second input lag on Firefox 149.** No matching bug found. Direct candidate for a new filing.
- **AT&T email login hangs in Firefox only.** No matching bug found. Same.

## Why this matters

These reports came from public posts that BSR doesn't capture. The bsky.app and youtube.com clusters span months. The matcher pass tells you in 2 seconds whether each item is "this is the bug we already have" or "this is something we don't seem to be tracking."

The platform-level reports (cert errors, HTTP Basic Auth, Android load failures) are the kind of thing that drives users to switch browsers. Catching them within hours of posting matters more than catching them at all.

## Cost and burden

- One-time backfill: ~$0.70 (classification + bug-match for three months of history)
- Ongoing: estimated $5–10/month at hourly cadence with prompt caching
- Maintenance: configuration tweaks (subreddit list, watchlist domains) take minutes; the source adapters are isolated, so adding Mastodon or Mozilla Connect later is one new file each
- Runs locally on my Mac via launchd; reports land in my Obsidian vault
- No credentials needed for Reddit, HN, or Bluesky (anonymous endpoints). Optional GitHub PAT to raise webcompat-bugs lookup rate limit.

## How to read the data

The reports live in three places, each suited to a different audience:

- **My Obsidian vault** (source of truth): `30 Resources/social-scanner/YYYY/YYYY-MM/YYYY-MM-DD.md`
- **A private GitHub Pages site** (for sharing): synced from the vault via `social-scanner share`. Easy to send a link to anyone with read access on the repo.
- **One-shot Google Doc exports** (for specific share-outs): a future addition; manually copy-paste for now.

Three days that show the format and value:

- [2026-04-24](2026/2026-04/2026-04-24.md) — biggest single-day cluster, includes the verbatim drive.google.com match
- [2026-04-26](2026/2026-04/2026-04-26.md) — includes the bsky.app cluster
- [2026-04-29](2026/2026-04/2026-04-29.md) — mostly platform regressions

Each item carries: source link, posted timestamp, sentiment, severity, urgency score, gist, reproduction steps when available, and bug cross-references grouped by match verdict (Likely match / Maybe related / Same domain different issue).

## What I'd do next

In rough order of value:

1. **Triage the 57 no-match webcompat items + 13 no-domain items.** Walk through the dated reports, file new Bugzilla bugs where appropriate. Highest-urgency ones (YouTube RAM, FF Basic Auth, Facebook Marketplace lag) are the obvious starting points.
2. **Decide what to do with umbrella matches.** Bugs like "bsky.app - site is not usable" attract many user reports describing different specific symptoms (login broken, 403 errors, mobile not loading). Worth deciding whether those should be split into specific issues.
3. **Run for two weeks unattended** to confirm the hourly cadence is stable and the daily Sonnet TL;DR pass produces useful synthesis.
4. **Add Mastodon and Mozilla Connect** as sources. Both are friendly APIs and likely yield more concentrated WebCompat signal than Bluesky's general feed.
5. **Bugzilla integration**: a "file from triage view" button so flagged items become bugs without copy-paste.
6. **Shared dashboard** for the WebCompat triage team if any of the above proves out. v1 is single-user; the data model and adapters are designed to make multi-user a deployment concern, not a rewrite.
